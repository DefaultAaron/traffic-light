## 概述

第二阶段第一轮训练在第一阶段（3类颜色）基础上扩展到 **7 类方向性检测**（red / yellow / green / redLeft / greenLeft / redRight / greenRight）。R1 的目标：

1. 在现有合并数据集（S2TLD + BSTLD + LISA 的方向重标注子集）上验证 7 类训练可行性，取得基准指标。
2. 打通从训练到 Orin 端 TensorRT C++ 推理的端到端链路。
3. 识别性能瓶颈并为 R2 制定改进方案。

**训练硬件**：RTX 4090 D（远程 GPU 服务器）  
**部署硬件**：NVIDIA Jetson AGX Orin 64GB（JetPack 5.1.2 / TRT 8.5.2）  
**框架**：Ultralytics（AGPL-3.0）

## 关键问题 — Demo 视频表现不佳

在 `demo/demo.mp4` 上的 C++ 推理结果：**多数帧无检测**，偶有检测时**边界框明显过大**。详细排查见下。

### 诊断 1：引擎输入分辨率与宣称不符（高优先级）

- `runs/detect/yolo26n-r1/weights/best.onnx` 的输入张量形状：`[1, 3, 640, 640]`
- 对应 `.engine` 同样是 640×640 固定输入
- 但 `inference/demo.py` 和 `inference/cpp/src/demo.cpp` 默认 `imgsz=1280`
- C++ 管线检测到不一致后**回退到引擎尺寸**并打印 `[TRT] warning`（`trt_pipeline.cpp:296-300`）— 实际推理仍在 640 分辨率

**影响**：`main.py export` 未显式指定 `--imgsz`，默认沿用训练时的 640。小目标在 640 输入下仅剩 6–20 像素，在未匹配训练域的视频上几乎无法识别。

### 诊断 2：Demo 视频与训练数据严重域偏移

| 维度 | 训练数据 | Demo 视频 |
|------|---------|-----------|
| 分辨率 | BSTLD 1280×720 / LISA / S2TLD | **544×960** |
| 方向 | 横屏 dashcam | **竖屏手机** |
| 宽高比 | 1.78 | **0.57** |
| 视角 | 车载固定位 | 手持 |

Letterbox 到 640×640：竖屏内容缩放为约 363×640，剩余 ~43% 面积为灰色填充；小目标进一步压缩。

### 诊断 3：数据本身的小目标偏置 + 类别失衡

- 训练标签 >80% 宽度 < 3%（imgsz=640 下 <20 px）→ 模型学到的"典型检测尺寸"极小；域外泛化时少数高置信度响应常落在大面积区域（非交通灯），造成"框过大"错觉。
- 模型位置先验集中在上右四分之一（dashcam 倾向），竖屏视频违反该先验。
- `redRight` / `greenRight` 类别各仅 10–20 条样本，R1 模型对这两类实际上没学到。

### 诊断 4：早停触发过早

`patience=20` 在类别极不均衡下可能把小样本类别还未收敛的曲线判为"无改善"。YOLO26n-r1 在第 19 轮达到 mAP50-95=0.485 后震荡至第 39 轮停止。

### 诊断 5：验证集同域

`runs/detect/yolo26n-r1/val_batch*_pred.jpg` 显示验证集全部是 `bstld_*.png` 横屏 dashcam — 指标健康但**未覆盖部署域**（竖屏 / 手机 / 非欧美路况）。目前指标不能外推到实际场景。

**结论**：性能差距是 **推理管线配置错误 + 域偏移 + 数据本身缺陷** 三因素叠加，非单纯数据质量问题。

## R1 诊断实验（2026-04-21，Ultralytics 原生推理）

绕开 TRT / ONNX / Head-strip 等部署链路的所有环节，直接用 `model.predict()` 在原始 `.pt` 权重上复现"检测稀疏 + 框过大"现象，定位故障源头。命令：

```bash
uv run python -c "
from ultralytics import YOLO
m = YOLO('runs/detect/yolo26s-r1/weights/best.pt')
m.predict('demo/demo.mp4', imgsz=1280, conf=0.25, save=True,
          project='runs/diagnose', name='s-pt-1280', device='mps')
"
```

### 结论一：**"框松动"源自低推理分辨率，但 C++ demo 中看到的显著"框过大"另有来源**

取 demo 视频 f100（右侧绿灯清晰可见）为诊断基准点，单帧多分辨率对比（`conf=0.10` 阈值，YOLO26s-r1）：

| imgsz | 检测数 | 最高置信度 | 边界框 w×h（原图像素，544×960）| 相对面积 |
|-------|-------|-----------|-----------------------------|---------|
| 640 | 1 | 0.281 | 28×82（框比真实灯体高 ~2×）| 0.44% |
| 1280 | 1 | 0.214 | 20×49 | 0.19% |
| **1536** | 1 | **0.579** | **19×43**（紧贴目标）| 0.16% |

- 在 640 上，`.pt` 预测框比真实灯体拉长了 ~80%，但**仍小于 1% 的画面面积**，并非 C++ demo 里肉眼可见的"框过大"。
- 在 1280 / 1536 上，框尺寸稳定在 ~20×45，与目标像素基本吻合。
- 置信度不随分辨率单调上升：640→1280 略降（0.28→0.21），但 640→1536 翻倍（0.28→0.58）。即 **1536 是本 demo 的最佳推理尺寸**。
- C++ demo 中观察到的显著"框过大"在 `.pt` 原生推理中**不能复现**，因此定位在 C++ 管线（见结论四 + 结论三末段）。

### 结论二：**YOLO26s-r1 支持跨尺度推理；YOLO26n-r1 不支持**

同一 f100 输入，两个模型对比：

| 模型 | 640 | 1280 | 1536 |
|------|-----|------|------|
| YOLO26n-r1 | **conf=0.52**, 24×66 | 0 detections | 0 detections |
| YOLO26s-r1 | conf=0.28, 28×82 | conf=0.21, 20×49 | **conf=0.58**, 19×43 |

- n-r1 在训练分辨率（640）上表现优异（conf 0.52，框紧贴），但一旦超出 640 就完全失效 — 容量不足导致尺度泛化差。
- s-r1 在 640 置信度较低但在 1536 上显著跃升 — 具备跨尺度泛化能力。
- **结论**：部署必须用 s-r1，不能用 n-r1。如果要节省延迟，宁可降低 `--imgsz` 也不要换到 n 模型。

### 结论三：**检测稀疏主要来自视频内容，而非模型失败**

对全视频 5,964 帧每 60 帧采样一次（共 100 帧），所有分辨率下 Ultralytics `.pt` 推理结果：

| 配置 | 有检测的帧占比 | 累计检测数 | 最大边界框面积占比 | 异常大框 (>20% 面积) |
|------|--------------|-----------|-------------------|-------------------|
| n @ 640 | 3.0% | 4 | 1.8% | 0 |
| n @ 1280 | 2.0% | 3 | 0.3% | 0 |
| n @ 1536 | 1.0% | 1 | 0.3% | 0 |
| s @ 640 | 2.0% | 2 | 0.6% | 0 |
| s @ 1280 | **5.0%** | 5 | 0.4% | 0 |
| s @ 1536 | **5.0%** | 6 | 0.3% | 0 |

- demo 视频前 150 帧内包含车辆入场经过红绿灯区域的片段，之后长时间行驶在无交通灯的工业试车场 — "5% 有检测"与视频内容一致，不是模型故障。
- s 在 1280 / 1536 的检测帧数约为 640 的 2.5 倍，证实提高推理分辨率对 s-r1 有实质收益。
- n 在 1280 / 1536 上检测数逐级递减（4→3→1），佐证结论二（n 不能跨尺度）。
- **所有分辨率下，`.pt` 直接推理均未出现>20% 面积的异常大框**。因此此前在 C++ demo 中观察到的"边界框明显过大"现象，**并不是模型或训练数据的缺陷，而是 C++ TRT 管线在 engine（640）与请求 imgsz（1280）不一致时的回退路径伪影** — 重建 1280 引擎后预期自然消失。
- 箭头类（redLeft / greenLeft / redRight / greenRight）全程未出现过检测 — 本 demo 场景可能不含箭头灯，无法据此判断箭头检测能力。

### 结论四：**部署链路存在未对齐的默认值**

- `runs/detect/yolo26{n,s}-r1/weights/best.onnx` 和对应 `.engine` 输入均固化为 **640×640**（导出时未传 `--imgsz` 致沿用训练分辨率）。
- `inference/demo.py` 默认 `imgsz=1280`、`inference/cpp/src/demo.cpp` 默认 `imgsz=1280`。
- C++ 端（`trt_pipeline.cpp:296-300`）检测到不一致后静默回退到引擎尺寸 640，仅打印 `[TRT] warning`。
- **实际 C++ demo 一直在 640 推理**，落入结论一的"框过大"模式。

### 同域 val 分辨率扫描（YOLO26s-r1，MPS）

```
imgsz=640:  mAP50=0.8497  mAP50-95=0.6073
imgsz=1280: mAP50=0.7023  mAP50-95=0.4359
```

- 同域（横屏 dashcam）val 集上，**更高推理分辨率反而降低 mAP50 约 14.7 pp**。
- 原因：模型在 640 训练，内部特征图尺度（stride 8 / 16 / 32 对应 80×80 / 40×40 / 20×20）和匹配到训练目标的先验尺寸都锁定在 640。在 1280 推理时，真实灯体被放大到 2× 像素宽但小目标先验未相应缩放，FP 增加、TP 减少。
- **关键推论**：部署端提高 `--imgsz` 到 1280 / 1536 对**同域数据会降低质量**，但对**竖屏 / 非欧美 / 手机 OOD demo**有收益（结论一 / 三）— 两者指向相反方向。
- → R1 的"1280 引擎 hotfix"**不是全局最优**，而是"以牺牲同域精度换取 OOD 能用性"的短期方案。长期方案必须在 R2 以 `imgsz=1280` 直接训练（见 R2 计划 §6）。

### 决策表

| 假设 | 证据 | 结论 |
|------|------|------|
| 部署管线 bug（引擎固化在 640 + C++ 端 imgsz 不匹配静默回退）| ✅ 确认：`best.onnx` 输入 `[1,3,640,640]`；`.pt` 原生推理无"过大框" | **主因** |
| 数据/域偏移导致无法泛化 | ❌ 部分否定：s-r1 在 1536 上能识别 demo 真实绿灯（conf 0.58）| 次要 |
| 训练分辨率不足 | 部分肯定：s-r1 在 640 置信度低于 1536，但差距可由部署端 imgsz 补偿 | 次要 — 部署端可缓解 |
| 模型容量不足 | 模型相关：n-r1 不能跨尺度，s-r1 可以 | 仅影响 n — 部署必须选 s |

**主导因素：引擎固化在 640×640 与 C++ 管线在 engine/requested imgsz 不一致时的回退路径伪影。其他问题的暴露都是这两个 bug 的二阶放大。**

## 第二轮（R2）计划

R2 目标：**关闭部署域与训练域的差距**，而非继续堆模型容量。

### 先做：部署侧立刻可修复的项（不依赖重训）

1. **在 imgsz=1280 重新导出 ONNX 并重建引擎**：  
   `uv run python main.py export runs/detect/yolo26s-r1/weights/best.pt --format onnx --imgsz 1280` → 裁头 → Orin `trtexec --fp16`。先验证 R1 权重在 1280 下的 demo 表现，隔离"引擎分辨率"这一变量。
2. **搭建部署域评估集**：从实际部署摄像头采集 100–300 帧，标注 7 类，作为独立 `data/eval_deployment/`。所有后续对比以该集为准，不再只看同域 val。

### 数据侧：R2 前必须完成

3. **扩充方向重标注子集**（延续 P1 行动计划第 5 项）：
   - BSTLD 测试集 8,334 张 → 方向重标注（优先级：高）
   - S2TLD `normal_2` 4,168 张 → 方向重标注
   - S2TLD `normal_1` 796 张 → 方向重标注
4. **右转箭头数据补齐**：
   - 方案 A（短期）：利用 `fliplr` 反向增强将 `redLeft`/`greenLeft` 合成为 `redRight`/`greenRight`，同时翻转图像和标签类别映射。
   - 方案 B（长期）：在国内道路场景采集包含右转箭头的真实数据。
5. **评估"去除极稀疏类"的备选方案**：若 R2 前无法把 `redRight`/`greenRight` 各自补至 >500 条，考虑合并为 `rightArrow` 单一类或暂时移出训练目标。

### 训练侧

6. **`imgsz=1280` 训练**（而非 640）：与部署侧对齐，同时减轻小目标信息损失。预估训练时间 3–4×，显存占用相应增加（可通过 `batch=-1` 自动调节）。
7. **`patience=40`**：小样本类别收敛更慢，放宽早停。
8. **类别加权 / 过采样**：
   - 用 Ultralytics `cls_pw` 或自定义 sampler 对 `yellow`、`greenLeft` 过采样 2–3×。
   - `redRight` / `greenRight` 视 §5 决定。
9. **域适应性增强**：
   - 启用 `degrees=5.0`（小角度旋转，模拟手机抖动）、`perspective=0.0005`。
   - 增加 JPEG 压缩 / motion blur 等真实相机伪影增强（需自定义 callback 或 albumentations 管线）。
   - **不开启** `fliplr` — 箭头方向会与标签不一致（除非配合方案 A 的类别同步翻转）。
10. **主力模型**：YOLO26s（R1 下降幅度最小，性价比最优），YOLO26m 作为上限基准并行训练。YOLO26n 在 R2 暂缓（容量不足问题在 R1 已证实）。

### 验收标准（R2 退出条件）

R2 若满足以下全部，方可进入 R3 / 定型：

- [ ] 部署域评估集上 mAP50 ≥ 0.60（7 类整体）
- [ ] 黄灯召回率 ≥ 0.70（不依赖于过采样下的高 precision 陷阱）
- [ ] 方向类（redLeft / greenLeft）mAP50 ≥ 0.55
- [ ] Demo 视频上主观检测率（人工抽查）≥ 80%，无明显过大框误报
- [ ] Orin 端 1280 × 1280 引擎 FP16 延迟 ≤ 50 ms / 帧

## R2 范围扩展（PM 确认事项）

2026-04-21 PM 会议新增两项诉求。两者均为"加类别"类型的扩展，与现有 R2 计划可在**同一次训练**中完成 — 不会额外引入轮次，但改变数据准备的优先级和规模。

### 扩展一：栏杆 / 道闸检测（新增 2 类）

新增类别：**`armOn`**（栏杆抬起 / 放行）、**`armOff`**（栏杆放下 / 阻挡）。

- **架构可行性**：仅把 `nc` 从 7 提到 9；YOLO26 共享主干，新头通道增量占模型 FLOPs <3%，延迟可忽略。
- **数据可行性**：S2TLD / BSTLD / LISA **均不含栏杆**，必须从零采集或外源获取。
  - 首选：**部署现场拍摄**（与部署域评估集合并采集，一次出差覆盖两件事）。
  - 备选：Roboflow Universe（"boom gate" / "railway barrier" / "parking barrier"）— 商业许可须逐条核对，否则只能用于预训练不可部署。
- **数据预算**：每类 **≥500 实例** 为最低线（R1 `redRight`=19/`greenRight`=13 是反面教材）；建议 5–15K 实例 / 类，与 `redLeft` (13K) 同量级，避免把 `green`/`red` (46–52K) 的平衡进一步恶化。
- **类别定义必须先敲定再标注**：`armOn` 是"抬起=通行"还是"arm 本身存在（有栏杆）"？下游消费者（PLC / 决策层）如何使用这两个信号？重标注成本极高，建议在标注启动前出一页 SOP 锁定定义。

### 扩展二：交通灯类别待扩展（7 → 最多 12，未定）

PM 表示最终类别数在 **7 到 12 之间** 取决于部署场景观测结果。候选方向（需等 PM 清单）：

- 行人信号：`pedRed` / `pedGreen` / `pedYellow`（若部署场景有行人过街）
- 方向类补全：`yellowLeft` / `yellowRight`
- 黄闪警示、熄灯、故障状态

**架构可行性**：与扩展一同理，`nc` 改到 9–12 是纯数据问题。

**数据可行性（关键）**：

| 类别来源 | 标注难度 | 说明 |
|---------|---------|------|
| 已有数据集中可重标注的（如从现有 `yellow` 细分出 `yellowLeft`）| 低 | 仅需脚本 + 人工复核 |
| 已有数据集中**不存在**的（如 `pedRed`）| 高 | 须新采 / 外源，同栏杆 |
| 夜间 / 恶劣天气的既有类别 | 中 | LISA 部分覆盖 |

如果 PM 最终决定的类别**全部能从现有素材中获得**（如从 `yellow` 派生 `yellowLeft`），成本可控；如果有新语义（如行人信号），成本与栏杆任务同级。

### 合并策略：一个 9–12 类模型，而非多模型

理由：
1. **共享主干的正则效应**：9–12 类联合训练通常比分两个模型的结果更鲁棒；栏杆和交通灯语义差异大，互相干扰低。
2. **部署成本**：Orin 64GB 显存冗余充足，但一个引擎只跑一次推理，延迟只付一遍。若拆两模型，1280×1280 下的 ~25 ms/帧 → ~50 ms/帧，逼近 50ms 预算上限。
3. **维护成本**：一份 ONNX → 一份引擎 → 一套后处理 → 一份 class map。双模型意味着以上乘 2。

**唯一例外**：若栏杆任务与交通灯任务在**不同摄像头 / 不同更新频率**下工作（例如摄像头 A 只看栏杆、摄像头 B 只看红绿灯），则拆分更合理。此点须 PM 确认部署拓扑。

### 对既有 7 类交通灯的影响评估

| 维度 | 风险 | 缓解 |
|------|------|------|
| 特征空间竞争 | 低 — 栏杆（水平杆状）与信号灯（方形灯组）特征显著不同 | 无需额外动作 |
| 类别不均衡加剧 | 中 — 若栏杆数据 >50K 实例会压制黄灯、方向类 | 栏杆数据上限 15K / 类 |
| 训练时间 | 小增 — head 额外 2 (或 5) 通道 | 可忽略 |
| 推理延迟 | 可忽略 | — |
| 验收标准牵连 | 中 — 同一模型的 mAP 会被新类别 dilute | R2 验收按类别分项看，不看整体 mAP |

### 对 R2 数据计划的优先级重排

在 §数据侧已有的 3–5 项基础上，**追加** 两项并与现有项共享出差机会：

1. **现场采集专项**（合并三目标）：
   - 部署域评估集（既定）
   - `armOn` / `armOff` 训练 + 评估数据
   - 若 PM 清单包含行人信号等新类，一并采集
2. **PM 清单锁定**（阻塞项）：在数据采集开始**之前**出一页 SOP，锁定最终类别清单 + 每个类别的视觉定义 + 边界情况处理（如"半升起的栏杆" / "灯熄灭"怎么归类）。SOP 缺失是标注返工的第一原因。
