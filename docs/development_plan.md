# 交通信号灯识别 — 开发计划v1.0

## 项目概述

为自动驾驶系统开发交通信号灯检测与状态分类模块，部署于 **NVIDIA Tegra Orin (64GB)** 平台。该平台同时运行完整的自动驾驶软件栈（感知、规划、控制、传感器融合等），因此信号灯模块必须轻量化，尽量减少 GPU/内存占用。

---

## 一、候选模型

| 模型 | 参数量 | COCO mAP | 免NMS | 定位 |
|------|--------|----------|-------|------|
| **YOLO26-n** | ~2.5M | 40.9% | 是 | 主力候选（最轻量） |
| **YOLO26-s** | ~9M | 47.5% | 是 | 主力候选（精度更优） |
| YOLO26-m | ~20M | 51.5% | 是 | 备选（需验证资源占用） |
| **YOLO11-n** | ~2.6M | 39.5% | 否 | 主力候选（稳定基线） |
| **YOLO11-s** | ~9M | 43.5% | 否 | 主力候选 |
| YOLO11-m | ~20M | 50.4% | 否 | 备选 |
| **RT-DETR-L** | ~32M | ~53.0% | 是 | 仅作精度上限参考，不作部署 |

### 选型说明

- **YOLO26**：最新版本（2026年1月），免NMS端到端推理，延迟更稳定；STAL（小目标感知标签分配）有助于远距离信号灯检测
- **YOLO11**：成熟稳定，社区资源丰富，作为对比基线
- **RT-DETR-L**：Transformer架构，精度最高，但参数量过大（~32M），不适合部署，仅作精度天花板参考
- 所有模型均通过 Ultralytics 统一训练框架，确保公平对比

## 三、开发阶段

### 第一阶段：基础颜色检测（当前）

**目标**：检测信号灯并分类为红/黄/绿三类

| 类别ID | 类别 | 说明 |
|--------|------|------|
| 0 | `red` | 红灯（含所有方向变体） |
| 1 | `yellow` | 黄灯（含所有方向变体） |
| 2 | `green` | 绿灯（含所有方向变体） |

**类别映射规则**：
- 所有方向变体合并为基础颜色（如 `stopLeft` → red，`goLeft` → green）
- 过滤 `off`、`wait_on` 等非可操作状态
- 三个数据集全部参与训练

**工作内容**：
1. 编写数据转换脚本（S2TLD/BSTLD/LISA → 统一 YOLO 格式）
2. 合并数据集，80/20 分层划分训练/验证集
3. 训练 5-7 个模型变体（YOLO26 n/s/m, YOLO11 n/s/m, RT-DETR-L）
4. 对比评估：mAP、逐类精度、推理延迟、GPU 占用、模型大小
5. 选定最优模型架构与尺寸

**训练策略**：

- 使用 COCO 预训练权重进行迁移学习（非从零训练）
- ~100 epochs，基于验证集 mAP 早停
- 禁用垂直翻转（`flipud=0.0`），保留 HSV 增强
- 合并数据集统一训练（非逐数据集顺序训练，避免灾难性遗忘）

### 第二阶段：方向检测 + 栏杆检测（R2，当前正在准备）

**目标**：在 R1（7 类）基础上扩展为**联合检测模型**，同时输出交通灯（9–12 类）与道路栏杆（1–2 类），共 **10–14 类**。

> 本阶段范围的最终确认请参见 `docs/phase_2_round_1_report.md` §"R2 范围扩展（PM 确认事项）"。下表汇总当前已锁定部分与待 PM 决定部分。

#### 2.1 交通灯类别（最少 9，最多 12）

**R1 已有 7 类 + R2 确认新增 2 类 = 最少 9 类：**

| 类别ID | 类别 | 状态 | 说明 |
|--------|------|------|------|
| 0 | `red` | R1 已有 | 红色圆灯 |
| 1 | `yellow` | R1 已有 | 黄色圆灯 |
| 2 | `green` | R1 已有 | 绿色圆灯 |
| 3 | `redLeft` | R1 已有 | 红色左转箭头 |
| 4 | `greenLeft` | R1 已有 | 绿色左转箭头 |
| 5 | `redRight` | R1 已有 | 红色右转箭头 |
| 6 | `greenRight` | R1 已有 | 绿色右转箭头 |
| 7 | `forwardGreen` | **R2 确认新增** | 绿色直行箭头（国内灯组常与圆灯独立设置） |
| 8 | `forwardRed` | **R2 确认新增** | 红色直行箭头 |

**至多可再新增 3 类（总计 ≤12），最终清单由 PM 确认**，候选包括：
- 行人信号灯（`pedRed` / `pedGreen` / `pedYellow`）
- 黄色方向变体（`yellowLeft` / `yellowRight` / `forwardYellow`）
- 闪烁 / 故障状态

> 前置核查：BSTLD / LISA / S2TLD 现有样本中疑似存在将直行箭头误标为 `red` / `green` 的情况；正式采集新数据之前，先用重标注脚本扫一遍确认缺口规模。

**设计原则**：
- 检测模型只负责识别**看到了什么**（颜色 + 灯型），规划模块决定**怎么做**。圆灯与箭头的含义组合由规划模块消解（`green` + `redLeft` + `redRight` → 仅可直行 等）。
- 黄灯方向变体默认仍折叠为黄色圆灯，除非 PM 显式将其纳入新增 3 类。
- **S2TLD 标注说明**：原始 S2TLD 无方向标注且存在漏标。`Annotations-fix/` 为人工重标注版本，已补充方向标签（redLeft, greenLeft）。normal_1 和 normal_2 子集仍为原始 3 类标注。

**各数据集方向标注覆盖情况（R1 已整理）**：

| 数据集 | 圆灯 (R/Y/G) | 左转箭头 (R/G) | 右转箭头 (R/G) | 说明 |
|--------|--------------|----------------|----------------|------|
| S2TLD | 全部 → 圆灯 | redLeft (572), greenLeft (77) | 无 | 方向标注仅限 Annotations-fix（重标注版）；normal_1/normal_2 为3类 |
| BSTLD | 有 | RedLeft (1,092), GreenLeft (178) | RedRight (5), GreenRight (13) | 直行箭头原先折叠为圆灯；R2 需恢复为 `forwardRed` / `forwardGreen` |
| LISA | 有 | stopLeft (12,734), goLeft (2,476) | 无 | `goForward` (205) R1 折叠为 green；R2 改标为 `forwardGreen`；`warningLeft` → 黄色圆灯 |

#### 2.2 栏杆检测（最少 1 类，最多 2 类）

PM 在 2026-04-21 例会确认需在同一模型中加入道路栏杆检测。采用分级目标：

| 目标 | 类别 | 说明 |
|------|------|------|
| **MVP（必做）** | `barrier`（1 类） | 仅检测栏杆存在，不区分抬起/落下。+1 类。 |
| **最佳实践（条件达成）** | `armOn` / `armOff`（2 类） | 落杆 / 抬杆状态分别建模。**仅当采集数据每种状态 ≥500 实例且多样性足够时启用**；+2 类。 |

**采集策略**：
- 与部署现场实采同批进行（一次出差同时得到：部署评估片段 + 栏杆训练数据 + 候选新增灯型数据）。
- 初期按 MVP 单类标注；若数据充分支持两种状态，标注阶段再批量子标签（CVAT / LabelMe 批量改签）。
- 数据量目标：MVP 2–5K 实例；两类版本各类 1–5K，类间比例不超过 3×。**需低于 `red` / `green`（~50K）**，避免放大现有类别不平衡。

#### 2.3 总体规模

- **类别总数（nc）范围**：10（9 TL + 1 barrier，下限）至 14（12 TL + 2 barrier，上限）
- **模型形态**：**单一联合模型**（非多模型拼接）。理由：共享 backbone 正则化、Orin 在 1280 分辨率下仍有延迟余量（现 25 ms，预算 50 ms）、运维成本减半。
- **前置条件（阻塞项）**：PM 需先给出最终交通灯类别清单 + 一页标注 SOP（视觉定义、边界样本），否则不启动新数据采集。SOP 缺失是 R1 重标注的主因。

#### 2.4 代码改动点（类别清单锁定后一次性改）

每处改动都是琐碎的（修改常量 / 配置），难点是类别清单本身而非代码：

- `data/traffic_light.yaml`（`names` 列表）
- `scripts/strip_yolo26_head.py` 调用站点（`--num-classes` 参数）
- `inference/cpp/include/trt_pipeline.hpp`（`kClassNames`）
- `inference/cpp/src/demo.cpp`（`kColors`）
- `inference/trt_pipeline.py`（`CLASS_NAMES`）
- 任何下游 ROS 2 消息转换 / 后处理阈值配置

## 五、已知问题与风险

### 1. 数据标注问题

| 问题 | 详情 | 影响阶段 |
|------|------|----------|
| **方向标注不一致** | S2TLD 方向标注仅限原始子集；LISA 有左转；BSTLD 训练集有方向但稀缺，测试集（8,334张）无方向标注需重标注 | 第二阶段 |
| **右转箭头数据极度稀缺** | 仅 BSTLD 有右转标注（RedRight 5 条、GreenRight 13 条），需重标注或采集新数据 | 第二阶段 |
| **直行箭头数据待恢复** | R1 曾将直行箭头折叠为圆灯；R2 新增 `forwardRed`/`forwardGreen` 后需从 BSTLD/LISA 已有样本中恢复，并核查是否存在误标 | 第二阶段 |
| **栏杆无公开数据集** | `barrier` / `armOn`/`armOff` 需现场实采；两态模式要求每态 ≥500 实例且多样性足够 | 第二阶段 |
| **黄灯样本偏少** | S2TLD 黄灯仅 75 个标注（红灯 1235，绿灯 816）；LISA `warning` 仅 5.5K（`go` 99K） | 两个阶段 |
| **扩展类 SOP 未定** | PM 最终类别清单 + 标注 SOP 未发布；SOP 缺失是 R1 重标注的主因 | 第二阶段（阻塞） |

**商用化路径**：

  - 购买 Ultralytics 企业许可证，或基于 Apache-2.0 的框架重新实现
  - 仅使用 S2TLD（MIT）数据，或采集/购买商用授权数据集
  - 采集实际驾驶场景数据并标注（同时解决域差异问题）

### 3. 所需资源

| 资源 | 用途 | 优先级 |
|------|------|--------|
| **GPU 训练服务器** | 完整训练 5-7 个模型变体（M4 Pro 仅适合快速验证） | 高 — 第一阶段 |
| **Tegra Orin 开发板** | 部署测试、TensorRT 推理延迟实测、GPU 占用评估 | 高 — 第一阶段 |
| **Ultralytics 企业许可** | 商用部署（如选用 YOLO/RT-DETR Ultralytics 版） | 生产阶段 |
| **标注工具 + 标注人力** | 第二阶段方向标注补充、生产阶段自有数据标注 | 第二阶段 |
| **实际驾驶数据采集** | 商用数据集构建、覆盖中国路况信号灯样式 | 生产阶段 |

### 4. 技术风险

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 黄灯检测精度不足 | 黄灯样本远少于红/绿灯，可能影响召回率 | 数据增强（HSV 扰动）、过采样、关注 per-class 指标 |
| 远距离小目标漏检 | 信号灯在远处可能仅占几个像素 | YOLO26 STAL 机制、copy-paste 增强、评估 AP-small |
| Orin 资源超标 | 模型 GPU 占用超出预算（<5% 目标） | 优先选用 nano 变体、TensorRT FP16 优化 |
| 数据集域差异 | 三个数据集来自不同国家/场景，可能与中国路况差异较大 | 评估模型在实际场景的表现，必要时采集本地数据微调 |
